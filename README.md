# 래더 자동 생성 R&D (탐색 기반 회로 합성)

> git 비추적 영역 (`/docs` gitignore). 진행 기록은 이 파일에서 관리.

## 한 줄 정의

래더 회로를 사람이 직접 설계하는 대신, **원하는 동작(타임차트 스펙)만 주면
컴퓨터가 탐색으로 회로를 찾아내게 할 수 있는가** — 그 타당성을 작은 규모에서
실증하는 연구.

## 왜 하는가 (제품 연결)

현재 Synex 래더 export는 사람이 짠 템플릿 기반 (`manual/create_*_ladder`,
1X01/1X21 등). 새 회로 유형마다 빌더 코드를 사람이 작성해야 한다.
성공 시 장기 방향:

- 타임차트 에디터 UI에서 입출력 동작만 그리면 회로 자동 생성
- 디바이스 목록은 기존 CM/Action 정의에서 자동 공급
- 종착점은 CEGIS 루프 (반례 자동 생성 ↔ 합성 반복)

## 시스템 구조 — 4계층

```
┌─ 탐색기 ──────── 회로 후보를 "생성"
│   random (베이스라인) / mcts (순수 UCT) / mcts_w (가중 롤아웃) / gp (유전 프로그래밍)
├─ 스펙 ────────── "정답"을 정의
│   타임차트 시나리오 + don't-care(타이밍 관용) + invariant(전역 금지 속성)
├─ 시뮬레이터 ──── 후보를 "채점" (보상 함수의 심장)
│   스캔 단위 PLC 실행 모델 (자기유지, TON, PLS, SET/RST)
└─ IR ──────────── 래더 미니 언어 (Contact/And/Or/Timer/Pulse/Coil)
```

| `ladder_benchmark.py` | 난이도 사다리 8과제 + don't-care 마스킹 + 퇴화 가드 + 4방법 비교 러너 |

실행 (import는 플랫 — **이 폴더 안에서** 실행, 한글 출력은 `PYTHONUTF8=1` 필요):

## 난이도 사다리 8과제

1. `self_hold` — 자기유지 (조합+피드백)
2. `interlock` — 전후진 인터로크 (출력 2개 상호참조, 선입력 우선)
3. `one_shot` — 원샷 (PLS)
4. `delayed_off` — 지연 정지 (TON+자기유지)
5. `flicker` — 플리커 (TON 2개 교차결합)
6. `seq2` — 2단 시퀀스 (래치 체인, ref 크기 14)
7. `actuator` — 전후진 실린더 (지령 latch + 센서 해제 + 인터로크, self_hold×interlock, ref 14)
8. `seq3` — 3단 시퀀스 (seq2 +1단, ref 22 — GP 한계 탐침)

방법론: 과제마다 사람이 짠 레퍼런스 해 → 기대 trace를 simulate로 도출 →
intent assert로 의도 검증 (마스킹 전 raw spec에서) → don't-care 마스킹.
완벽해 판정은 `accuracy==1.0 AND invariant 위반 0` (score는 페널티 섞여 부적합).

## 실험 일지

### 2026-06-10 — 파이프라인 구축 + 순수 UCT 반증

- IR/시뮬레이터/스펙/무작위 베이스라인 구축. 자기유지를 무작위 16k~43k에 발견
- 최초 "MCTS 4~40배" 결론은 **판정 기준 버그** (`score≥0.99`가 크기 페널티
  포함이라 MCTS에 유리). accuracy/score 분리 후 공정 재측정
- **순수 UCT는 무작위에 전패.** 패인: (a) 기만적 부분점수 가지에 예산 집중
  (b) 균등 롤아웃의 PUSH 지배 (c) 분기 14~25 × 깊이 10~18에서 얕은 prefix만 기억

### 2026-06-11 오전 — don't-care 결착

- `mask_transition_scans(width=1)`: 입력 전이 스캔 기대값을 None 치환
  (invariant는 마스킹 스캔에도 적용 — 타이밍 관용이지 전역 속성 면제가 아님)
- **함정 1 (퇴화):** 초기 seq2 시나리오가 전이 위주 압축이라 61% 마스킹 →
  "항상 꺼진 회로"가 acc 1.0 통과. → dwell 스캔 연장 + `assert_spec_not_degenerate`
  (항상0/항상1 회로 차단 가드) 추가
- **함정 2 (width=2 금지):** interlock 0.940 고원의 정체는 `Y0=X0` 꼼수가
  차단 의도 채점 1점만 희생하는 구조. width=2면 그 1점마저 마스킹돼 꼼수가
  정답 판정됨. → 시나리오 C dwell 연장 + 대칭 시나리오 D 추가로 대응
- **결과:** delayed_off 무작위 211~1,784회 (마스킹 전 5k~46k, **~25배**) —
  과잉명세 가설 실증. interlock/seq2는 잔존 실패 → 순수 탐색 한계로 재분류

### 2026-06-11 오후 — 탐색 개선 1차 (3종 구현+측정)

| 과제 | random | mcts | mcts_w |
|---|---|---|---|
| delayed_off | 3/3 (211~1,784회) | 0/3 (0.833) | **2/3 (336 / 5,191회)** |
| seq2 (SET/RST) | 0/3 (최고 0.910) | 0/3 (최고 **0.974**) | 0/3 (최고 **0.974**) |
| interlock (invariant) | 0/3 (0.933) | 0/3 (0.933) | 0/3 (0.933) |

1. **가중 롤아웃 (`weighted_rollout`) — 실증.** 균등 샘플의 PUSH 지배(10:2)를
   종류 우선 가중 샘플로 해소 → **MCTS 계열 최초 성공** (delayed_off 336회)
2. **SET/RST EMIT 액션 — 부분 실증.** `BuildState(allow_setrst)` +
   `GenCfg.setrst_p`. seq2 rung 한도 4 (SET/RST 래치 체인 자연형이 4 rung).
   seq2에서 MCTS 계열이 무작위 최초 추월 (0.974 vs 0.910), 발견은 실패
3. **invariant — interlock 가설 반증.** `Spec.invariants` + `evaluate()` +
   score 위반당 -0.05. 9런 전부 0.933 그대로. 원리적 이유: invariant 페널티는
   트리 탐색의 보상 지형만 바꾸고, **무작위 샘플링은 매 시도 독립이라 무영향**.
   MCTS도 꼼수 분지 억제만으로는 크기 8 교차결합 2 rung을 조립 못 함.
   invariant 자체는 정답 판정 정합성(trace 일치해도 동시 ON이면 불합격) 위해 유지

### 2026-06-11 저녁 — GP 도입, 난이도 사다리 전 과제 결착

`ladder_gp.py` 신설 — rung 목록 단일점 교차 + 변이 5종(서브트리 교체/접점
반전/코일 교체/rung 추가·삭제) + 토너먼트 k=3 + 엘리트 적합도 캐시 +
새 피 5%. 교차 후 Timer/Pulse 이름 재부여(부모 간 상태 공유 차단).
생성기는 benchmark의 `random_program_ext`/`random_logic_ext` 주입 재사용.

| 과제 | random | mcts | mcts_w | gp |
|---|---|---|---|---|
| interlock | 0/3 (0.933) | 0/3 (0.933) | 0/3 (0.933) | **3/3 (1,148 / 5,087 / 14,643회)** |
| seq2 | 0/3 | 0/3 (최고 0.974) | 0/3 (최고 0.974) | **3/3 (16,890 / 91,151 / 116,550회)** |

- seq2 발견 해는 SET/RST 코일을 실제로 사용 — 액션 공간 확장(오후)과
  GP(저녁)가 **합성되어** 풀린 사례. 크기 14 과제까지 200k 내 해결
- 이로써 **난이도 사다리 6과제 전부 어떤 방법으로든 발견 가능** 상태 도달

### 2026-06-11 밤 2 — 이중 코일 금지 (실무 호환 구조 제약)

발견 해 검토 중 "자기유지를 같은 디바이스 OUT 2개의 덮어쓰기 드모르간
변형으로 발명"한 사례 확인 — 동작은 맞지만 GX Works3 이중 코일 경고
패턴이라 사람이 읽기 혼란. 구조 제약으로 금지하기로 결정.

규칙 (`coil_allowed`, GX Works3 기준): **OUT은 디바이스당 유일 작성자,
SET/RST는 OUT과 혼합 금지** (SET+RST 래치 쌍·다중 SET은 허용).
적용 지점: 무작위 생성기(후보 필터) / GP(교차 후 `dedupe_coils` 수선 +

### 2026-06-11 밤 3 — 스펙 보존 축소 (해 정리 완결)

구문적 단순화의 한계(seq2 38 vs ref 14 — 런타임 상관관계로 무의미해진
항은 구문만으로 증명 불가) 확인 후, **스펙을 oracle 로 쓰는 greedy
ablation** (`shrink_program`, delta debugging 원리) 추가. 후보 변형 =
rung 삭제 / And·Or 인자 삭제 / Timer·Pulse 벗기기 — `evaluate()` 가
변하지 않을 때만 채택 (구조상 스펙 보존 보장, 크기 단조 감소로 종료).
표준 절차 `polish_program` = 단순화 → 축소 → 마무리 단순화.

| 과제 | GP 발견 | 단순화(동작 보존) | **축소(스펙 보존)** | 레퍼런스 |
|---|---|---|---|---|
| seq2 | 58 | 38 | **15** | 14 |
| delayed_off | 39 | 20 | **9** | 9 |
| flicker | 11 | 9 | 9 | 8 |

- seq2 축소 해가 `X0 -> SET Y0; X1 -> RST Y0` + Y1 rung — **사람이 쓰는
  래치 관용구 그대로** 드러남. delayed_off 는 레퍼런스와 동일 크기 9

변형이 학습 가능해짐).

## 핵심 교훈 누적

1. **문제의 절반은 스펙 언어 설계.** 과잉명세(don't-care로 해소) ↔ 과소명세

   무거운 도구는 무작위·개선 MCTS가 둘 다 무너지는 지점에서만 정당화
3. **판정 기준·채점 분포의 버그가 결론을 뒤집는다.** 모든 비교는 동일 예산
